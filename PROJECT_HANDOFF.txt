Security Posture After This Pass
================================
Improved
--------
1. Tokens are no longer sent in web-client query strings.
2. Tokens are no longer persisted in browser localStorage.
3. Admin endpoints fail closed in production if GROCERY_ADMIN_TOKEN is missing.
4. Admin token comparison uses constant-time comparison.
5. API responses include basic browser security headers:
   - X-Content-Type-Options: nosniff
   - X-Frame-Options: DENY
   - Referrer-Policy: strict-origin-when-cross-origin
   - Permissions-Policy: geolocation=(), microphone=(), camera=()
   - Cache-Control: no-store
6. Auth and expensive endpoints have lightweight in-memory rate limiting.

Still Not Production-Grade
--------------------------
The current auth model is email-only login. Anyone who knows an existing user's email can
request a token. This is acceptable only for a prototype or private demo.

Before real users:
1. Replace email-only login with passwordless email links, OAuth, or username/password with
   password hashing and reset flows.
2. Move rate limiting to a shared store such as Redis if running more than one API process.
3. Put the API behind HTTPS only.
4. Set strict production CORS origins.
5. Use real observability and alerting.

Production Environment Checklist
================================
Set these environment variables in production:

1. GROCERY_ENV=production
2. GROCERY_ADMIN_TOKEN=<generate a long random secret>
3. GROCERY_API_CORS_ORIGINS=https://your-real-frontend-domain.example
4. GROCERY_DB_PATH=<persistent database path>
5. GROCERY_TOKEN_TTL_MINUTES=<shorter TTL such as 1440 for one day, if appropriate>
6. GROCERY_RATE_LIMIT_ENABLED=true

If live pricing partners are used:
1. LIVE_PRICING_FLIPP_KEY=<partner key>
2. Any other provider-specific keys from config/live_pricing/providers.json
